# Music Production Skills

This guide maps music-production skills to Ableton MCP tools. It is written for agents that need to turn a musical brief into a finished Ableton production without guessing, overclaiming, or skipping safety gates. For the step-by-step language interpretation workflow, start with [Natural language to music](NATURAL_LANGUAGE_TO_MUSIC.md).

Current tools are implemented by the MCP server today. Future tools are roadmap items from [Future patches](FUTURE_PATCHES.md) and must not be reported as available until `npm run verify:mcp` lists them. The current producer-brain layer is summarized in [Producer brain](PRODUCER_BRAIN.md).

## Golden Agent Loop

Use this loop for serious production work:

```text
brief
-> readiness and control-mode check
-> create production session
-> song blueprint
-> signature sound palette
-> production assets
-> dry-run execution plan
-> render review
-> focused revision pass
-> delivery readiness
```

Use `ableton_create_production_session` first when a client needs fewer calls and persistent state. Use the lower-level loop below when debugging, changing one subsystem, or inspecting why the facade chose a recommendation:

```text
brief
-> readiness and control-mode check
-> concept plan
-> source and sample selection
-> tempo, key, groove, and motif decisions
-> layer stack and device-chain plan
-> arrangement and automation plan
-> dry-run execution
-> user approval for real writes
-> render or export plan
-> audio analysis
-> focused revision pass
-> release readiness check
```

Do not try to solve the full track in one call. Good agents make one production decision at a time, inspect the result, then revise.

## Source Mode First

Before sample or source-heavy work, choose a source usage mode:

- `private_experiment`: unverified sources may be used for private iteration, but the agent must record them in a source manifest as `unverified` or `experiment_only`.
- `release_candidate`: unverified or experiment-only sources become warnings or blockers in `ableton_check_release_source_readiness`.

This changes release readiness only. Downloads, Ableton writes, UI/mouse control, and remote HTTP remain explicitly gated.

## Agent Operating Rules

- Start with read-only readiness unless the user explicitly asks for a narrow offline artifact.
- Keep `dry_run=true` until a plan is reviewed.
- Require `ABLETON_MCP_ENABLE_WRITE=1` and `dry_run=false` for real Ableton writes.
- Require `ABLETON_MCP_ENABLE_DOWNLOADS=1` for downloads or staged imports from remote sources.
- Require `ABLETON_MCP_ENABLE_UI_CONTROL=1` for foreground UI/mouse control.
- Prefer background LiveAPI bridge control over UI control.
- Use named UI actions before raw coordinates.
- Do not broadly scan user folders, AppData, browser profiles, credentials, or unrelated music folders.
- Treat remote sample metadata as untrusted until filtered and staged.
- Preserve attribution sidecars through delivery.
- When the bridge reports `unsupported: true`, stop or choose a documented fallback.
- When a live call times out, report setup steps instead of claiming success.

## Do Not Do This

- Do not pile layers without named roles.
- Do not write before dry-run review.
- Do not package unverified or experiment-only sources as release-ready.
- Do not use UI/mouse control unless the user explicitly chooses it.
- Do not claim LiveAPI success when the bridge returns unsupported or times out.
- Do not master over arrangement problems.
- Do not keep adding effects when level, timing, or note choice is the real issue.
- Do not copy a reference song's protected melody, lyrics, arrangement, or vocal identity.
- Do not create hidden commands, coercive speech, or unclear vocal messages.

## Recommended Deepening Order

For the tools already listed in [Producer brain](PRODUCER_BRAIN.md), the next work is deeper DSP, stored plan state, better scoring, or proven LiveAPI execution. For the remaining roadmap-only tools, implement in this order because each layer improves the next:

1. Tempo and groove: deepen `ableton_plan_tempo_grid`, then add `ableton_generate_groove_map` and `ableton_score_timing_feel`.
2. Musical writing: deepen `ableton_generate_harmonic_palette`, `ableton_generate_motif_system`, and `ableton_score_hook_memorability`, then add voicing, bassline, and counterline tools.
